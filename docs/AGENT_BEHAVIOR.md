# Agent Behavior

이 문서는 에이전트가 코딩, 문서화, 검토 작업을 할 때 지켜야 하는 행동 기준입니다. 특정 제품이나 기술 스택을 정하지 않고도 적용되는 기준만 둡니다.

## Source

이 기준은 `multica-ai/andrej-karpathy-skills` 저장소의 네 가지 원칙을 이 하네스 스타터에 맞게 재구성한 것입니다. 원본은 Claude Code용 단일 지침과 skill 형태이며, 라이선스는 MIT입니다.

원본 저장소: <https://github.com/multica-ai/andrej-karpathy-skills>

## Principle 1. Clarify Before Committing

애매한 요구를 조용히 해석해서 진행하지 않습니다.

- 가정이 있으면 먼저 명시합니다.
- 해석이 여러 개면 가장 단순한 해석을 기본값으로 두되, 위험한 선택이면 질문합니다.
- 보안, 데이터 손실, 외부 서비스 비용, 되돌리기 어려운 아키텍처 변경은 확인 없이 진행하지 않습니다.
- 불확실한 부분은 `TBD`, `blocked`, `needs_verification`처럼 보이는 상태로 남깁니다.

## Principle 2. Prefer The Smallest Viable Change

요청을 만족하는 가장 작은 변경을 먼저 선택합니다.

- 한 번만 쓰는 추상화는 만들지 않습니다.
- 미래 확장을 이유로 런타임, 프레임워크, 패키지, 설정을 추가하지 않습니다.
- 제품 범위가 미정이면 구조를 채우지 않고 결정 기준만 기록합니다.
- 변경이 200줄이 될 수밖에 없다면, 먼저 더 작은 산출물로 나눌 수 있는지 확인합니다.

## Principle 3. Keep Changes Traceable

모든 변경은 사용자 요청, 문서화된 결정, 검증 기준 중 하나로 추적 가능해야 합니다.

- 주변 코드를 겸사겸사 정리하지 않습니다.
- 기존 스타일과 문서 구조를 우선합니다.
- 내가 만든 미사용 항목은 정리하지만, 기존 죽은 코드나 오래된 문서는 요청 없이 삭제하지 않습니다.
- 관련 없는 문제를 발견하면 고치기보다 별도 후속 작업 후보로 기록합니다.

## Principle 4. Work From Verifiable Goals

완료 기준은 느낌이 아니라 재현 가능한 증거입니다.

- 버그 수정은 재현 조건과 검증 명령을 남깁니다.
- 기능 추가는 관찰 가능한 사용자 행동과 수용 기준을 둡니다.
- 문서 작업은 단일 진실 원천, 갱신된 상태, 다음 세션 인계 가능성을 확인합니다.
- 검증하지 못했다면 완료로 표시하지 않고 남은 위험을 적습니다.

## Trade-Offs

이 기준은 단순한 오타 수정이나 명백한 한 줄 변경에서는 다소 느릴 수 있습니다. 대신 비자명한 작업에서 잘못된 가정, 과한 추상화, 관련 없는 변경, 검증 없는 완료 선언을 줄이는 데 초점을 둡니다.

## Starter-Specific Completion Check

작업을 마치기 전에 다음을 확인합니다.

- 변경 범위가 이번 요청과 연결됩니다.
- 새 결정은 적절한 원천 문서에 기록됐습니다.
- `feature_list.json` 상태가 실제 진행과 맞습니다.
- 가능한 검증을 실행했고, 결과 또는 미검증 사유가 남았습니다.
- 다음 세션이 추가 설명 없이 이어받을 수 있습니다.
