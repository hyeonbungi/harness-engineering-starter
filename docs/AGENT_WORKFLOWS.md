# Agent Workflows

이 문서는 Codex 같은 에이전트를 실제 프로젝트에서 어떻게 운영할지 정리하는 기준입니다. 특정 제품, 모델, 기술 스택, 실행 환경을 확정하지 않고도 적용되는 작업 흐름만 둡니다.

## Source

이 기준은 Avid의 X article `How to Master Codex in 2026 (Builder's Course)`에서 확인한 운영 패턴을 이 하네스 스타터에 맞게 재구성한 것입니다. 해당 글은 실무 경험 기반 참고 자료이며, OpenAI 제품 사양의 공식 원천은 아닙니다.

## Operating Model

에이전트 작업의 단위는 채팅 한 번이 아니라 검증 가능한 작업입니다.

- 작은 작업은 한 세션에서 끝냅니다.
- 큰 작업은 계획, 범위, 제외 사항, 검증 방법, 중단 조건을 먼저 둡니다.
- 산출물은 답변 자체가 아니라 검증된 변경, 리뷰 가능한 diff, 인계 가능한 문서입니다.
- 진행 중 판단이 바뀌면 조용히 계속하지 않고 상태와 근거를 남깁니다.

## Surface Selection

작업 표면은 "얼마나 가까이 지켜볼 것인가"로 고릅니다.

| Surface type | Use when | Starter rule |
| --- | --- | --- |
| Local interactive session | 사용자가 변경을 지켜보고 바로 피드백해야 함 | 기본값 |
| Headless/scripted run | 반복 명령, 정적 점검, 단발 조사 | 명령과 결과를 남김 |
| Browser or GUI surface | UI 동작, 시각 상태, 폼 흐름을 확인해야 함 | `DESIGN.md`와 시각 증거를 함께 사용 |
| Async/cloud-like work | 독립 작업을 맡기고 나중에 리뷰해도 됨 | 제품과 검증 루프가 안정된 뒤 사용 |
| Automation | 반복되는 유지보수 작업이 검증됨 | 실패/중단 조건과 권한을 먼저 정의 |

## Good Task Briefs

좋은 작업 지시는 네 요소 중 필요한 것을 포함합니다.

| Element | Meaning |
| --- | --- |
| Outcome | 원하는 결과와 완료 기준 |
| Example | 입력, 출력, 화면, 실패 사례 같은 구체 예시 |
| Constraints | 사용할 파일, 경계, 금지 사항, 성능/보안 조건 |
| Reminder | `AGENTS.md`, `DESIGN.md`, 제품 명세, 기존 패턴 |

모든 작업에 네 요소가 다 필요하지는 않습니다. 단순 수정은 결과와 경계만으로 충분할 수 있고, 새 기능이나 구조 변경은 네 요소를 모두 요구합니다.

## Single-Change Rule

하나의 프롬프트에는 하나의 주요 변경만 둡니다.

- 버그 수정, 테스트 추가, 리팩터링, 문서 갱신을 한 번에 섞지 않습니다.
- 같은 목표 안에서도 공유 파일을 건드리는 작업은 직렬화합니다.
- 병렬화는 서로 독립된 파일, 모듈, 조사 범위에만 사용합니다.
- 30초 안에 끝나는 작은 확인 작업은 병렬화하지 않습니다.

## Plan-First Rule

다음 작업은 구현 전에 짧은 계획을 먼저 둡니다.

- 제품 동작이나 사용자 여정이 바뀝니다.
- 여러 파일, 모듈, 문서 원천을 건드립니다.
- 검증 실패 시 되돌림 경로가 필요합니다.
- 보안, 권한, 데이터, 외부 서비스 비용에 영향을 줍니다.

계획은 길 필요가 없습니다. 목표, 범위, 제외 사항, 검증 방법, 중단 조건이 보이면 충분합니다.

## Failure Behavior

검증이 실패하면 테스트나 오류를 숨기지 않습니다.

- 실패한 명령과 관찰 결과를 기록합니다.
- 테스트를 통과시키기 위해 테스트 기대값을 임의로 낮추지 않습니다.
- 예외를 삼키는 코드로 실패를 가리지 않습니다.
- 원인이 불명확하면 수정 전에 가설과 확인 방법을 먼저 남깁니다.

## Skills And Reusable Workflows

같은 지시를 반복해서 쓰게 되면 skill 후보로 기록합니다.

초기 후보는 다음 세 가지입니다.

- `investigate`: 재현, 관련 경로 확인, 원인 가설, 검증 계획까지 수행하고 수정은 보류합니다.
- `new-feature`: 제품 명세와 기존 패턴을 읽고 계획, 구현, 검증, 인계까지 이어갑니다.
- `finish-pr`: 브랜치, 검증, 커밋, PR 본문, 리뷰 요청을 표준화합니다.

이 스타터에서는 바로 skill 디렉터리를 만들지 않습니다. 실제 프로젝트에서 같은 워크플로가 반복될 때 `.agents/skills/` 또는 개인 skill로 승격합니다.

## Parallel Work

병렬 작업은 속도보다 격리를 기준으로 판단합니다.

- 조사 범위가 넓고 결과만 요약하면 되는 경우에 적합합니다.
- 독립 기능이나 독립 문서 묶음에는 적합합니다.
- 같은 파일, 같은 상태, 같은 DB migration을 건드리면 직렬화합니다.
- 작성한 에이전트와 검증하는 에이전트는 가능하면 분리합니다.

## Verification Ladder

완료 신뢰도는 아래 순서로 올립니다.

1. 정적 검증: JSON, 포맷, 타입, 린트, 문서 baseline
2. 실행 검증: 테스트, 빌드, smoke check, 재현 명령
3. 행동 검증: 브라우저, 스크린샷, UI flow, 운영 로그
4. 독립 검토: fresh reviewer, 다른 모델/도구, 사람 리뷰
5. 인계 증거: `feature_list.json`, `claude-progress.md`, `docs/session-handoff.md`

컴파일 성공은 충분한 완료 증거가 아닙니다. 사용자 행동이 바뀌는 작업은 행동 검증이나 그에 준하는 관찰 증거가 필요합니다.

## Tool And Context Hygiene

도구가 많을수록 좋은 것이 아닙니다.

- MCP, plugin, connector는 실제로 쓰는 것만 켭니다.
- 읽기 도구와 쓰기 도구의 권한을 구분합니다.
- 비밀값과 개인 토큰은 문서나 설정 예시에 넣지 않습니다.
- 파괴적 명령, DB write, force push, 외부 비용 발생 작업은 확인과 되돌림 계획이 필요합니다.
- 긴 참고 자료는 루트 지침에 붙이지 않고 `docs/references/`로 분리합니다.
