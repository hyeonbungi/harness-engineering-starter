# 0003. Agent Behavior Guardrails

- Status: Accepted
- Date: 2026-05-25

## Context

이 스타터는 아직 완성된 제품 scaffold가 아니며, 새 프로젝트가 추측 없이 시작되도록 돕는 하네스입니다. 따라서 에이전트 지침도 단순히 "무엇을 읽을지"를 넘어서, 모호함을 다루는 방식, 변경 범위, 검증 기준을 일관되게 강제해야 합니다.

`multica-ai/andrej-karpathy-skills`는 LLM 코딩 에이전트가 흔히 저지르는 잘못된 가정, 과한 추상화, 관련 없는 수정, 약한 완료 기준 문제를 네 가지 원칙으로 정리합니다. 이 방향은 현재 스타터의 목적과 맞지만, 원문을 그대로 붙이면 프로젝트 중립성, 한국어 문서 체계, `feature_list.json` 기반 상태 관리와 어긋날 수 있습니다.

## Decision

원본 지침을 그대로 복사하지 않고, 하네스 스타터의 문서 체계에 맞춘 `docs/AGENT_BEHAVIOR.md`를 둡니다.

- 모호한 요구는 가정과 위험을 명시한 뒤 진행합니다.
- 가장 작은 유효 변경을 선호합니다.
- 모든 변경이 요청, 결정, 검증 기준으로 추적되게 합니다.
- 완료 선언은 검증 증거를 요구합니다.
- 출처와 라이선스는 `docs/references/karpathy-guidelines.md`에 기록합니다.

루트 `AGENTS.md`와 `CLAUDE.md`는 이 문서를 시작 순서와 완료 기준에 포함합니다.

## Consequences

- 장점: 에이전트 행동 기준이 루트 지침에만 흩어지지 않고 별도 원천 문서로 관리됩니다.
- 장점: 기술 스택이 없어도 적용 가능한 품질 기준이 생깁니다.
- 장점: 과한 구현, 추측 기반 결정, 관련 없는 diff를 줄일 수 있습니다.
- 단점: 단순 작업에서도 확인 항목이 늘어날 수 있습니다.
- 단점: 지침 문서가 하나 늘어나므로 `init.sh`와 진행 문서를 함께 관리해야 합니다.

## Alternatives

- 원본 `CLAUDE.md`를 그대로 붙인다: 빠르지만 이 저장소의 한국어 하네스 문서 구조와 중복됩니다.
- 루트 `AGENTS.md`에만 추가한다: 파일 수는 줄지만 지침이 길어지고 재사용성이 떨어집니다.
- 별도 적용하지 않는다: 현재 규칙만으로도 최소 운영은 가능하지만, 에이전트 행동 품질을 평가하기 어렵습니다.

## Rollback

이 기준이 과하다고 판단되면 `docs/AGENT_BEHAVIOR.md` 내용을 루트 지침의 짧은 체크리스트로 축소하고, 이 결정 기록에는 superseded 사유를 남깁니다.
